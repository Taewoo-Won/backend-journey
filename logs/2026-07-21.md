## 1. 발단 — “체크가 안 되어 있다”

로드맵 페이지(taewoo-won.github.io/backend-journey)에서 체크 표시가 비어 보인다는 관찰에서 시작했다. 이게 며칠간 흐릿하게 맴돌던 질문 — **“자바 기초는 도대체 언제 완결나냐”** — 의 정체였다. 로드맵의 어느 칸이 비어 있으니 기초가 안 끝난 것처럼 느껴진 것.

**추측 말고 실물 확인.** roadmap.json을 직접 열었다:

```
root@ledger:~/backend-journey# grep -E '"id"|"status"' roadmap.json

          "status": "done",
          "id": "m102",
          "status": "in_progress"     ← ★ 여기
          "id": "m103",
          "status": "done"
          "id": "m104",
          "status": "done",
          "id": "m105",
          "status": "done"
          "id": "m106",
          "status": "done"
          "id": "m107",
          "status": "todo"
```

**m106은 정상 done이었다.** 어제(7/20) 처리한 게 제대로 박혀 있었다. 진짜 범인은 그 위 — **m102만 `in_progress`로 끼어 있었다.** m101 done → m102 진행중 → m103~106 done. 순서대로 쭉 done이다가 자바 기초 칸 하나만 비어서, 로드맵 전체가 미완성으로 보인 것.

```

          "status": "in_progress"
```

제목이 하필 “Java 기초”라, 이 칸이 비어 있는 게 정확히 “자바 기초 미완”으로 읽혔다.

## 2. 핵심 판별 — 실제 미완이 아니라 표시 누락이다

여기서 중요한 건 “m102를 다시 공부해야 하나?“가 아니라 **“이게 진짜 안 끝난 건가, 표시만 안 된 건가”**를 가르는 것이다. 논리로 증명된다:

- m103·m104·m105·m106이 **전부 done**이다.
- 이것들의 정체: m103(Spring 전환), m104·m105(예외·REST API), m106(소켓).
- 이 마일스톤들에서 실제로 한 일 — record 만들기, 예외 던지고 잡기, Map으로 계좌 관리, 제네릭 `ResponseEntity<T>` 쓰기 — 은 **전부 m102의 기초(변수·조건·클래스·메서드·리스트·맵·예외)를 딛고 선 것**이다.

> **뒤 단계가 done이라는 것 자체가, 그 앞 단계를 이미 관통했다는 증거다.** m102를 건너뛰고 m103~106을 할 수는 없다.

즉 m102는 mini-ledger 만들 때 이미 한참 전에 관통했는데 **done 체크만 누락**된 것. m106이 7/19에 겪었던 상태 불일치(진행은 됐는데 roadmap.json 갱신 누락)와 똑같은 종류다.

## 3. 조치 — m102 done 처리

```
root@ledger:~/backend-journey# sed -i '/"id": "m102"/,/"status"/ s/"status": "in_progress"/"status": "done"/' roadmap.json && grep -A3 '"id": "m102"' roadmap.json
          "id": "m102",
          "title": "Java 기초 (변수·조건·클래스·메서드·리스트·맵·예외)",

          "status": "done"
```

sed 범위 치환(`/"id": "m102"/,/"status"/` = m102 id 줄부터 그 다음 status 줄까지의 범위 안에서만 in_progress→done)이라 다른 마일스톤은 안 건드린다. JSON 무결성:

## 4. 매듭 — “자바 기초 완결?“에 대한 답

이 체크 하나로, 며칠 맴돌던 질문이 문자 그대로 닫혔다.

**답: 이미 완결됐고, 표시만 안 됐던 거다.** 실제 미완이 아니었다. 그런데 한 걸음 더 — 애초에 **“자바 기초 완결”을 별도 목표로 세우는 것 자체가 이 로드맵의 설계와 안 맞는다**는 걸 짚어둘 값어치가 있다:

|“자바 다 떼고 → 백엔드” (X)|“프로젝트하며 자바를 집어 씀” (이 로드맵)|
|-------------------|-------------------------|
|자바 = 먼저 완성할 독립 과목  |자바 = 프로젝트에 딸려오는 재료       |
|문법 챕터를 끝내야 다음      |필요한 문법을 그때그때 습득          |
|“기초 완결”에 종료선 있음    |별도 챕터 없음 — m102는 초반 관문일 뿐|

record는 mini-ledger에서, 애노테이션은 ledger-api에서, 소켓은 Redis에서 배웠다. **“자바 기초”라는 독립 챕터가 로드맵에 없는 이유가 이거다.** 그러니 앞으로 만날 자바(스트림, 동시성, 컬렉션 심화 등)는 m102의 “기초”가 아니라 **각 프로젝트에 딸린 심화**다. “기초 완결” 질문은 m102 체크와 함께 닫힌다.

**불안의 정체도 정리됐다:** “기초가 덜 됐다”는 느낌은 실제 미완이 아니라 ① 로드맵의 표시 누락(오늘 해결) + ② 배운 걸 인출 못 하는 불안(→ 이건 문법을 더 쌓을 게 아니라, 11일 밀린 복기 큐로 굳힐 일)이었다.

## 7. 회고

며칠 흐릿하던 질문의 정체가 로드맵 칸 하나였다. “자바 기초 언제 완결나냐”는 게, 알고 보니 m102 체크 누락 때문에 그 칸만 비어 보인 거였다. 실물(roadmap.json)을 열어보니 답이 명확했다 — m106은 멀쩡히 done이고, m102만 in_progress로 끼어 있었다. 추측으로 “기초가 부족한가” 자책할 게 아니라 파일을 열어 확인하는 것, 그게 먼저였다.

판별의 핵심은 논리였다. m103~106이 done이라는 것 자체가 m102를 이미 딛고 섰다는 증거다 — 기초 없이 그 위를 지을 수 없으니까. 이건 표시 누락이지 실제 미완이 아니라는 결론이 거기서 나왔다. m106이 7/19에 겪은 것과 같은 종류의 상태 불일치였고, 같은 방식(sed로 정정)으로 닫혔다.

그리고 더 큰 매듭 하나 — “자바 기초 완결”이라는 목표 자체가 이 로드맵엔 없다는 것. 자바는 먼저 떼고 쓰는 과목이 아니라 프로젝트하며 집어 쓰는 재료다. record도, 애노테이션도, 소켓도 각 프로젝트에서 필요할 때 배웠다. 그러니 “기초 완결”에 종료선을 그으려는 것 자체가 방향이 틀렸다. m102 체크와 함께 그 질문은 닫혔고, 남은 불안(배운 걸 인출 못 함)은 문법이 아니라 복기로 풀 일이다.

시간 없는 날이었지만 억지 빈 커밋 대신 실질 있는 정리 하나로 닫았다. 7/11·7/16과 같은 운영이다 — 로드맵이 실제 상태와 어긋나 있던 걸 바로잡은 거니, 이건 부풀린 잔디가 아니라 정당한 정리다. m101~106이 이제 끊김 없이 done으로 뜬다. 남은 건 월말 점검뿐, 자바 기초는 완료다.

> **한 줄:** “자바 기초 언제 완결?“의 정체는 m102 체크 누락이었다 — 실제 미완이 아니라 표시 누락(뒤 마일스톤이 done인 게 그 증거). sed로 정정→크론 자동 push→잔디 확정. m101~106 끊김 없이 done, 자바 기초 로드맵상 완료. 남은 불안(인출)은 문법이 아니라 복기로.
