# PART 1 — ★ 오늘 처음 만난 개념들 (풀코드 + 정밀 해설)

오늘은 mini-ledger(순수 자바)에서 스프링으로 넘어가는 전환기라 처음 보는 것이 많았다. **이 파트가 오늘 기록의 본체다.**

## 1-1. record — “그릇 클래스”를 한 줄로 만들어주는 자바 문법

**작성한 풀코드 (`TransferRequest.java` 전체):**

```java
package com.ledger.api;

public record TransferRequest(String fromId, String toId, long amount, String idempotencyKey) {}
```

**이게 뭔가.** `record`는 “데이터를 담기만 하는 클래스”를 선언 한 줄로 만들어주는 자바 문법이다. 저 한 줄을 컴파일하면 자바가 뒤에서 **자동으로 만들어주는 것**이 넷이다:

1. `private final` **필드 4개** (fromId, toId, amount, idempotencyKey)
1. 그 4개를 전부 받는 **생성자**
1. 각 필드를 꺼내는 **접근자** 4개
1. equals / hashCode / toString (덤)

**내가 이미 손으로 만들어본 패턴이다.** Transaction 클래스를 짤 때 정확히 이 구조를 전부 수동으로 쳤다. record 한 줄이 대체하는 것을 손으로 쓰면 이렇다:

```java
// ↓ record 한 줄이 대체하는 "손으로 쓴 버전" (Transaction을 짤 때 내가 쳤던 그 패턴)
public class TransferRequest {
    private final String fromId;
    private final String toId;
    private final long amount;
    private final String idempotencyKey;

    public TransferRequest(String fromId, String toId, long amount, String idempotencyKey) {
        this.fromId = fromId;
        this.toId = toId;
        this.amount = amount;
        this.idempotencyKey = idempotencyKey;
    }

    public String fromId() { return fromId; }
    public String toId() { return toId; }
    public long amount() { return amount; }
    public String idempotencyKey() { return idempotencyKey; }
    // + equals / hashCode / toString
}
```

→ 약 20여 줄이 괄호 안 선언 하나로 접힌다. 즉 **record는 새 개념이 아니라, 내가 아는 패턴(불변 데이터 그릇)의 자동화**다. 낯설게 느껴졌지만 실체는 이미 손에 익은 구조였다.

**★ 함정 — 접근자 이름이 게터와 다르다:**

```java
request.getFromId()   // ✗ record에는 이런 메서드가 없다 (컴파일 에러)
request.fromId()      // ✅ get 없이 "필드명 그대로 + 괄호"
```

게터 관례(getXxx)에 익숙해진 손이 그대로 나가면 바로 에러다. record는 **필드명 자체가 접근자 이름**이다. 컨트롤러 몸통에서 이 접근자로 값을 꺼낸다.

**왜 불변인가.** record는 만들어진 뒤 값을 못 바꾼다(setter가 없다). 요청 데이터가 처리 도중 바뀌면 안 되니 이게 맞다 — Transaction을 불변으로 설계했던 것과 같은 철학이 여기서도 통한다.

**왜 여기 들어갔나 — DTO라는 역할.** curl이 보내는 JSON(`{"fromId":"a",...}`)은 그냥 텍스트다. 이걸 자바 세계에서 다루려면 **타입 있는 그릇**이 필요하다. 이런 “요청/응답 데이터 전용 그릇”을 실무에서 **DTO**(Data Transfer Object)라 부른다. 도메인 객체(Bank, Account)와 달리 로직이 없고 **데이터 운반만** 한다 — 그래서 record가 딱 맞는 자리다. 오늘 만든 TransferRequest가 내 첫 DTO다.

## 1-2. @RequestBody + 잭슨 — JSON 텍스트가 자바 객체가 되는 마법 ④

**컨트롤러 시그니처:**

```java
@PostMapping("/transfer")
public String transfer(@RequestBody TransferRequest request) { ... }
```

**이게 뭔가.** `@RequestBody`는 “HTTP 요청의 **본문(body)**을 이 매개변수의 타입으로 **변환해서 꽂아라**“는 스티커다. 내가 파싱 코드를 한 줄도 안 썼는데, 메서드가 시작될 때 request에는 이미 완성된 TransferRequest 객체가 들어와 있다. 실제로 일어나는 일을 단계로 펼치면:

```
curl이 보낸 것:  {"fromId":"a","toId":"b","amount":1000,"idempotencyKey":"key-1"}   ← 그냥 텍스트
      ↓ ① 스프링이 요청 헤더 Content-Type: application/json 을 보고 "본문이 JSON이구나" 판단
      ↓ ② 잭슨(Jackson) — 스프링에 내장된 JSON 변환기 — 이 텍스트를 파싱
      ↓ ③ JSON의 키 이름(fromId)과 record의 필드명(fromId)을 짝지음
      ↓ ④ 사실상 new TransferRequest("a","b",1000,"key-1")를 대신 호출
transfer() 메서드 시작 시점에 request 매개변수에 완성된 객체가 꽂혀 있음
```

**★ 핵심 규칙: JSON 키 이름 = record 필드명.** 이 **이름 일치**가 매핑의 계약이다. 만약 JSON에서 `from_id`로 보내면 fromId와 짝이 안 맞아 그 자리는 null(참조형)/0(숫자형)으로 **조용히** 남는다 — 에러도 안 나고 값만 빠지는, 제일 잡기 어려운 유형의 버그다. 그래서 이름을 맞추는 게 규칙이 아니라 계약이다.

**왜 마법 목록에 넣나.** 스프링이 대신해주는 것 목록(어제 만든 것)에 오늘 하나가 추가된다:

```
① 내장 웹서버(Tomcat) 기동      — 어제
② 컴포넌트 스캔(@ 스티커 읽기)   — 어제
③ URL → 메서드 자동 연결        — 어제
④ JSON ↔ 자바 객체 자동 변환    — ★ 오늘 (잭슨)
```

이 넷이 “스프링이 나 대신 해주는 일”이고, 8월에 이 마법의 아래층(요청 받는 서버)을 직접 만들 때 이 목록이 지도가 된다.

## 1-3. @PostMapping — GET과 POST는 의미가 다르다

- **GET** (어제 /health): “**조회**한다”. 본문이 없다. `curl URL`만 치면 기본이 GET이다.
- **POST** (오늘 /transfer): “**생성/변경**한다”. **본문에 데이터를 실어** 보낸다.

**왜 transfer는 POST인가.** 이체는 서버의 상태(잔액)를 **바꾸는** 요청이다. HTTP에서 상태를 바꾸는 요청은 POST(류)로 보내는 게 의미론이다 — “조회는 GET, 변경은 POST”가 첫 원칙. (참고 지형: 수정 전용 PUT, 삭제 전용 DELETE도 있다. 지금은 GET/POST 둘이면 충분하고, 나머지는 만나면 그때.)

`@PostMapping("/transfer")` = “**POST /transfer** 요청이 오면 이 메서드를 실행하라.” 어제 @GetMapping의 POST판이다. 어제 GET /transfer로 쳤다면? — @PostMapping만 있으니 그 경로의 GET은 등록이 없어 405(Method Not Allowed) 계열로 거절된다. **경로뿐 아니라 방식까지 맞아야** 메서드에 도달한다.

## 1-4. curl POST — 플래그 세 개가 각각 하는 일

```bash
curl -X POST http://localhost:8080/transfer \
  -H "Content-Type: application/json" \
  -d '{"fromId":"a","toId":"b","amount":1000,"idempotencyKey":"key-1"}'
```

- **`-X POST`** — 요청 **방식** 지정. 안 쓰면 GET이다(어제 /health가 그랬다).
- **`-H "Content-Type: application/json"`** — “본문이 JSON”이라는 **라벨(헤더).** 1-2의 ①단계가 정확히 이 헤더를 본다. **이게 없으면** 스프링이 본문을 JSON으로 해석하지 않아 415(Unsupported Media Type) 계열로 거절된다 — 본문이 멀쩡해도 라벨이 없으면 안 읽는다.
- **`-d '...'`** — **본문(body)** 자체. 여기 실린 텍스트가 잭슨을 거쳐 record가 된다. (JSON 따옴표는 곧은따옴표여야 한다 — 폰 스마트 구두점 주의.)

어제 GET은 플래그 없이 한 줄이었는데 오늘 셋이 붙은 이유 = **“방식이 다르고(POST), 본문이 있고(-d), 그 본문의 형식을 알려야 해서(-H)”**다.

## 1-5. ★ 상태를 가진 컨트롤러 — 오늘의 개념 핵심

**작성한 풀코드 (`TransferController.java` 전체, 커밋 d0d6f9c):**

```java
package com.ledger.api;

import org.springframework.web.bind.annotation.PostMapping;
import org.springframework.web.bind.annotation.RequestBody;
import org.springframework.web.bind.annotation.RestController;

@RestController
public class TransferController {

    private final Bank bank = new Bank();

    public TransferController() {
        bank.createAccount("a", "이름", 2100);
        bank.createAccount("b", "이름", 3500);
        bank.createAccount("platform", "플랫폼", 0);
    }

    @PostMapping("/transfer")
    public String transfer(@RequestBody TransferRequest request) {
        bank.transfer(request.fromId(), request.toId(), request.amount(), request.idempotencyKey());
        return "a: " + bank.getAccount("a").getBalance()
             + ", b: " + bank.getAccount("b").getBalance()
             + ", platform: " + bank.getAccount("platform").getBalance();
    }
}
```

**줄 단위 해부 — 뭐가 왜 들어갔나:**

**`@RestController`** — 어제와 동일: 스프링이 부팅 때 컴포넌트 스캔(마법 ②)으로 이 스티커를 발견하고 “HTTP 담당 클래스”로 등록한다. 이 등록 덕분에 아래 @PostMapping이 URL 표에 오른다.

**`private final Bank bank = new Bank();`** — Bank를 **필드**로 둔다. 여기가 오늘의 개념 핵심이다.

- 컨트롤러 객체는 서버가 뜰 때 스프링이 **딱 한 번** 만들고, 서버가 사는 동안 유지한다. 따라서 이 bank도 **한 번 만들어져 계속 산다.**
- 그 결과: **요청이 100번 와도 전부 같은 bank를 쓴다.** 잔액·history·usedKeys가 **요청과 요청 사이에 이어진다.**

수명을 그림으로 보면:

```
서버 부팅
  └ 스프링 컴포넌트 스캔 → @RestController 발견
      └ new TransferController()  ← 딱 1회
          └ 생성자 실행: bank 생성 + 계좌 3개 등록  ← 딱 1회
요청 1 (key-1) ─┐
요청 2 (key-1) ─┼─▶ 셋 다 "같은 bank"를 씀 (필드라서)
요청 3 (key-2) ─┘
서버 종료(kill) → bank와 모든 상태 소멸
```

**Main 시절과의 결정적 차이:**

|       |mini-ledger (java Main)|ledger-api (서버)         |
|-------|-----------------------|------------------------|
|Bank 생성|**실행할 때마다** 새로         |부팅 때 **딱 한 번**          |
|상태 수명  |실행 끝나면 소멸              |**서버가 사는 동안 유지**        |
|두 번째 호출|첫 호출을 기억 못 함           |**첫 호출의 usedKeys가 살아있음**|

→ **오늘 검증이 이 개념의 증명이다.** key-1 2차 요청이 막힐 수 있었던 건, 1차 때 `usedKeys.add("key-1")`로 저장한 그 Set이 2차 요청 시점에도 **같은 bank 안에 살아있었기** 때문이다. Main 시절이었다면 실행마다 새 Bank라 막을 방법 자체가 없었다. “상태를 가진 서버”가 잔액 불변이라는 숫자로 눈앞에 증명된 것.

**한계 — 영속성이 없다 (인지만).** 이 상태는 **메모리에만** 있다. 서버를 재시작하면 a의 잔액도, 두 건의 history도, usedKeys도 **전부 증발**한다. “껐다 켜도 남는 기록”을 영속성(persistence)이라 하고, 지금은 그게 없다. 푸는 도구가 나중의 DB/JPA이며, **8월 Layer 0의 WAL이 정확히 이 문제**(디스크에 먼저 기록해 죽어도 남기기)를 다룬다. 오늘 “재시작하면 증발”을 직접 인지해둔 게 그때의 동기가 된다.

**생성자의 createAccount 3개** — 서버 부팅 시 1회 초기화다.

- a는 이체 1000 + 수수료 20 = 1020을 감당해야 하니 2100 (7/5 원자성 검증 `getBalance() < amount + fee`를 통과하도록).
- **platform 포함이 함정 회피 포인트** — Bank.transfer가 내부에서 `getAccount("platform")`을 부른다(수수료 입금). 빼먹으면 첫 이체가 AccountNotFoundException으로 터진다. 7/5에 Main에서 만난 그 함정을, 오늘은 기억해서 처음부터 넣었다.

**메서드 몸통 2줄:**

- ① `bank.transfer(request.fromId(), ...)` — request에서 record 접근자로 4개 값을 꺼내 도메인에 전달. Main이 `"a","b",1000,"key-1"`을 **직접** 쓰던 것과 유일한 차이가 이것 — **값이 이제 request 안에 있으니 꺼내서 넘긴다.**
- ② `return "a: " + ... getBalance() ...` — 세 계좌 잔액을 문자열로 조합해 반환. **return 값이 곧 HTTP 응답 본문**(어제 배운 것)이므로, m105 완료 조건(“잔액 변경 확인”)이 응답에 그대로 보이게 설계했다. mini-ledger Main에서 println으로 찍던 그 조합을 return으로 옮긴 것.

## 1-6. git rm –cached — 이미 커밋된 파일을 gitignore로 빼는 법

**상황.** 어제 server.log가 커밋에 딸려 들어갔다. .gitignore에 `server.log`를 추가했는데 — **그것만으론 안 빠진다.**

**메커니즘.** .gitignore는 “**아직 추적되지 않은** 파일을 새로 추적하지 말라”는 규칙이다. server.log는 이미 커밋돼 **추적 목록(인덱스)에 올라가** 있으므로 ignore가 무시된다. 그래서:

```bash
git rm --cached server.log
```

- `--cached` = “**추적 목록에서만** 제거하라” — 디스크의 실제 파일은 그대로 남는다(서버 로그는 계속 쌓여야 하니까).
- 추적에서 빠진 순간부터 .gitignore가 먹기 시작한다.
  → “이미 커밋된 파일을 ignore로 빼는” 표준 2단계: **ignore에 추가 + rm –cached.** (그냥 `git rm`이면 디스크 파일까지 지워진다 — cached가 핵심이다.)

## 1-7. sed ‘1i’ — 파일 첫 줄 앞에 삽입 (package 일괄 작업)

```bash
sed -i '1i package com.ledger.api;' Account.java
```

- `-i` = 파일을 **제자리에서** 수정
- `1i` = **1번 줄 앞에**(i = insert) 뒤의 텍스트를 삽입
- package 선언은 파일 **최상단**이어야 하므로 ‘1i’가 정확한 자리다.

여섯 파일에 for 루프로 일괄 적용한 것은, 어제 배운 규칙(“package = 폴더 경로와 일치”)의 자동화다. 손으로 여섯 번 열 일을 한 줄 루프로.

## 1-8. 부모·자식 프로세스 — gradle이 java를 “자식으로” 띄운다

`./gradlew bootRun &`로 서버를 띄우면 프로세스가 **둘** 생긴다:

```
[백그라운드 작업 %N] gradle(래퍼)  ← 부모
        └─ java(스프링 서버, 실제 8080 점유)  ← 자식
```

`[2] 1178776`(gradle)과 로그의 `PID 1178826`(java)이 달랐던 이유가 이것이다. `kill %2`는 부모(작업)를 죽이는데, 자식 java는 **시차를 두고** 정리되거나 드물게 남을 수 있다. → 그래서 서버를 내렸는지의 **판정 기준은 kill 명령이 아니라 `ss`의 빈 출력**이다(트러블 3-C에서 실제로 겪음).

**첫 시도에서 이미 맞았던 것 (7개):** 임포트 3줄 · @RestController · 클래스 골격 · `private final Bank bank = new Bank();` · @PostMapping 시그니처 · @RequestBody 매개변수 · **계좌 3개 초기화(platform 포함 — 7/5 함정을 기억해서 회피).** 즉 스프링 쪽 새 문법은 전부 첫 시도에 통과했다.

```java
bank.transfer(request.fromId(), request.toId(), request.amount(), request.idempotencyKey());
```

막힌 지점의 본질: **“값을 직접 쓰던 것 → request에서 꺼낸다”**는 한 뼘 차이. 그걸 넘으니 나머지는 아는 자바였다.

**교정 ② — return이 비어 있었다 (`return " ";`).** 그리고 `bank.getAccount("a").getBalance();` 한 줄은 결과를 아무 데도 안 써서 무의미했다(호출만 하고 버림). getBalance의 결과를 **return 문자열에 담아야** 응답에 잔액이 실린다:

```java
return "a: " + bank.getAccount("a").getBalance()
     + ", b: " + bank.getAccount("b").getBalance()
     + ", platform: " + bank.getAccount("platform").getBalance();
```

(println으로 찍던 조합을 return으로 — 서버에선 return이 곧 응답이므로.)

## 3-B. 404 Not Found — 진짜 원인은 “파일이 없었다”

**증상.** 서버 정상 기동(`Started ... in 3.789 seconds`) 후 첫 POST:

```
{"timestamp":"...","status":404,"error":"Not Found","path":"/transfer"}
```

**추적.** 서버는 떴고 compileJava도 SUCCESSFUL이었는데 404 = “**/transfer라는 경로가 서버에 등록돼 있지 않다**“는 뜻. 등록이 안 됐다면 컨트롤러가 스캔에서 빠졌다는 것 → 폴더를 직접 확인:

```bash
ls ~/ledger-api/src/main/java/com/ledger/api/
# → TransferRequest.java는 있는데 TransferController.java가 목록에 없다!
find ~/ledger-api -name "*ransfer*"
# → TransferRequest.java 뿐 — Controller 부재 확정
```

**원인.** **nano에서 짠 파일이 저장되지 않았다**(Ctrl+O 누락 추정). 파일 자체가 없으니 스캔할 대상도, /transfer라는 URL도 존재하지 않았다.

**여기서 배운 것 둘:**

1. **404의 정확한 뜻** = “그 경로가 등록돼 있지 않다.” 서버 다운(연결 거부)도, 코드 버그(500)도 아니고 — **경로의 부재.**
1. **“BUILD SUCCESSFUL”은 ‘있는 파일들’의 성공이지 ‘있어야 할 파일’의 존재 증명이 아니다.** 빌드 성공을 보고 파일이 있다고 **가정**했다 — 판정 순서는 파일 존재(ls/grep)부터. 이틀째 같은 문장이다: 가정 말고 확인.

**해결.** EOF heredoc으로 재작성(EOF는 저장 단계 자체가 없어 이 실수가 원천 차단된다) → 판정 `grep -c "PostMapping\|transfer\|RestController"` = **6** → `kill %1` 후 재기동 → `Started ... in 4.702 seconds` → 이후 200.

## 3-C. 서버 내림의 판정은 kill이 아니라 “ss 빈 출력”

**증상.** 검증 후 `kill %2` 직후 확인:

```
ss -tlnp | grep 8080
LISTEN ... users:(("java",pid=1178826,...))     ← java가 여전히 LISTEN?!
[2]+  Terminated  ... ./gradlew bootRun ...      ← gradle은 죽었는데
```

**전개.** 1-8의 구조 그대로 — kill %2는 **부모(gradle)**를 잡았고, **자식(java)**은 그 순간 아직 살아 보였다. `kill 1178826`을 치자 `No such process` — 이미 종료돼 있었다(부모 종료가 **시차**를 두고 자식까지 정리한 것; 첫 ss는 그 시차의 잔상). 재확인:

```bash
ss -tlnp | grep 8080     # → (빈 출력) = 포트 닫힘 확정
```

**교훈.** 공개 IP(104.236.232.210) 위 서버의 “내렸다” 판정 기준은 **kill 명령의 실행이 아니라 ss의 빈 출력**이다. 부모·자식 구조 때문에 kill 한 방이 즉시 안 먹을 수 있으니, 종료할 때마다 ss로 확인하는 것을 고정 절차로 삼는다.

**각 줄이 증명하는 것:**

- **1번** — JSON→record→Bank.transfer→잔액 return, 전 파이프라인이 산다. a가 1020 냄(이체+수수료), platform이 20 받음: 7/4~5의 정산·원자성 로직이 HTTP 아래서 그대로.
- **2번(하이라이트)** — 잔액이 **한 푼도 안 움직였다.** 1차 때 저장된 usedKeys(“key-1”)가 **같은 bank 안에 살아있어서**(1-5의 상태 유지) 2차를 초입에서 걸렀다. Main 시절엔 구조적으로 불가능했던 일이다.
- **3번** — 멱등성이 “중복만” 막고 “새 거래”는 통과시킨다는 반대편 증명. a: 1080→60 (또 1020).

**돈 보존 산수:** a 총 −2040 (1020×2) = b +2000 + platform +40. 두 이체 모두 온전 — 시스템 전체에서 돈이 새지도 생기지도 않았다.

**복식부기가 HTTP 뒤에서도 도는가:**

```bash
grep "돈 보존" ~/ledger-api/server.log
# 판정:
돈 보존됨 (합 = 0)
돈 보존됨 (합 = 0)          ← 정확히 두 줄
```

**★ “2”라는 숫자 자체가 구조 증명이다.** Bank.transfer의 첫 줄이 이거였다:

```java
if (usedKeys.contains(idempotencyKey)) { return; }    // 멱등성 관문 — 초입
...
checkBalance();                                        // 복식부기 — 맨 끝
```

→ 막힌 2번 요청은 **첫 줄에서 return**돼 checkBalance까지 아예 가지 않는다. 실제 이체된 1·3번만 끝까지 가서 “돈 보존됨”을 찍었다. 로그 줄 수 = 실제 실행된 이체 수. 어제까지 만든 원자성·복식부기가 HTTP 층 아래에서 **그대로, 정확한 횟수만큼** 살아 돈다.

**이게 왜 큰가.** `idempotencyKey`로 중복 요청을 흡수하는 이 구조는 **Stripe·토스 결제 API의 실제 계약** 그대로다. 네트워크에서 요청은 중복 전송될 수 있다(사용자 재시도, 타임아웃 후 자동 재전송) — 그때 이체가 두 번 되면 사고다. 6/29에 Set 하나로 개념으로 만든 것이, 오늘 “같은 키 두 번 = 잔액 불변”이라는 실전 검증을 통과해 **진짜 API 계약**이 됐다.

**(보너스 관찰 예약)** 지금 a=60이라 다음 1000 이체는 잔액 부족이다 — 시도하면 InsufficientBalanceException이 HTTP에서 **500**(서버 내부 오류)으로 샌다. 클라이언트 잘못(잔액 부족)인데 서버 탓(500)으로 표기되는 것 — **“자바 예외를 HTTP 상태코드로 어떻게 번역하나”**가 다음 조각이다. 오늘은 위치만 찍어둔다.

## 6. 개념 사전 (복기용 압축판 — 정밀 해설은 PART 1)

- **record** — 불변 데이터 그릇을 한 줄로. 필드·생성자·접근자 자동 생성. 접근자는 `fromId()`(get 없음).
- **DTO** — 요청/응답 데이터 운반 전용 그릇. 로직 없음. record가 적임.
- **@RequestBody** — 요청 본문(JSON)을 매개변수 타입으로 자동 변환해 주입. 잭슨이 수행. 매핑 계약 = JSON 키명 ↔ 필드명 일치.
- **@PostMapping** — POST 요청을 메서드에 연결. 조회=GET(본문 X), 변경=POST(본문 O). 방식까지 맞아야 도달.
- **curl POST** — `-X POST`(방식) / `-H Content-Type`(본문 형식 라벨, 없으면 415) / `-d`(본문).
- **상태를 가진 컨트롤러** — 스프링이 컨트롤러를 부팅 시 1회 생성 → 필드 bank가 요청 간 유지. 한계: 메모리뿐(재시작=증발, 영속성 부재 → DB/JPA·8월 WAL).
- **git rm –cached** — 추적 목록에서만 제거(디스크 보존). 이미 커밋된 파일에 ignore를 먹이는 표준.
- **sed ‘1i텍스트’** — 1행 앞 삽입. package 일괄에 사용.
- **부모·자식 프로세스** — gradle(부모)이 java(자식)를 띄움. 종료 판정은 ss 빈 출력.
- **404** — “그 경로가 등록돼 있지 않다.” 파일/스캔/매핑 부재를 의심.

## 10. 회고

오늘 가장 크게 남은 건 두 장면이다.

첫째, **도메인을 한 글자도 안 고치고 API가 됐다.** cp로 여섯 파일을 옮기고 package 한 줄씩 얹은 게 전부다. 6/19부터 Bank에 쌓은 검증·멱등성·복식부기가, 호출자를 Main에서 curl로 바꾸는 것만으로 HTTP 세계에서 그대로 돌았다. “로직을 Bank라는 깔때기에 모아두면 입구는 갈아끼울 수 있다”는 걸 말로만 알다가 오늘 몸으로 확인했다. 컨트롤러를 짜다 막힌 지점도 사실 스프링이 아니었다 — “값을 직접 쓰던 걸 request에서 꺼낸다”는 한 뼘 차이였고, 그걸 넘으니 나머지는 아는 자바였다. 짜기 시작할 때 “하나도 모르겠다”고 느꼈지만 뚜껑을 열어보니 골격 7개를 첫 시도에 다 맞혔더라. **낯섦과 모름은 다르다** — record도 그랬다. 처음 보는 문법인 줄 알았는데, Transaction을 30줄로 손수 짰던 바로 그 패턴의 자동화였다. 전환기의 공포는 대부분 이름이 바뀐 것에 대한 공포지, 내용이 바뀐 것에 대한 공포가 아니었다.

둘째, **key-1을 두 번째 보냈을 때 잔액이 안 움직인 순간.** 그 한 줄이 오늘의 하이라이트다. 서버 안의 Bank가 요청 사이에 살아있어서, 1차 때 저장한 usedKeys가 2차를 초입에서 막았다 — “상태를 가진 서버”라는 개념이 숫자 불변으로 눈앞에 증명됐다. Main 시절엔 실행마다 새 Bank라 이게 구조적으로 불가능했다는 걸 생각하면, 오늘 서버로 넘어온 의미가 정확히 이 한 줄에 있다. 그리고 이게 Stripe·토스가 쓰는 바로 그 패턴이라는 것 — 6/29에 Set 하나로 만든 멱등성이 오늘 실전 API 계약이 됐다. “돈 보존됨”이 정확히 2번 찍힌 것까지 — 막힌 요청은 검증까지 안 간다는 구조가 로그 줄 수로 드러난 게 묘하게 통쾌했다.

삽질도 셋 있었고 셋 다 판정 습관이 잡았다. IntelliJ가 먹통일 때 씨름 대신 nano로 우회한 것(진도가 우선, 도구 문제는 별도 과제 — 컴을 버린 게 아니라 오늘만 돌아간 것), 404에서 “빌드 성공 = 파일 존재”라는 가정이 깨진 것(ls부터가 순서였다), kill 후 ss로 재확인해 부모·자식 프로세스의 시차를 본 것. 이틀 연속 같은 문장이 값을 한다 — **가정 말고 확인.**

한계도 눈에 넣어뒀다. 지금 잔액은 메모리에만 있다. 서버를 껐다 켜면 a의 60원도, 두 건의 기록도, usedKeys도 전부 증발한다. 오늘은 그게 괜찮은 학습 단계지만, 진짜 원장은 껐다 켜도 남아야 한다 — 8월 WAL이 정확히 그 문제고, 오늘 이 “증발”을 직접 인지한 게 그때의 동기가 될 것이다.
