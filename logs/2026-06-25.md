# PART 1 — 왜 커스텀 예외인가 (결제의 본질)

## 1-1. IllegalArgumentException의 한계

지금까지 계좌가 없을 때도, 잔액이 부족할 때도 전부 `IllegalArgumentException` / `IllegalStateException`을 던졌다. 이건 호출자에게 딱 한 마디만 한다: **“뭔가 잘못됐어.”**

근데 **결제는 실패 경로가 곧 제품이다.** 실패했을 때 *왜* 실패했는지에 따라 완전히 다르게 대응해야 한다:

|실패 이유 |대응                    |
|------|----------------------|
|계좌가 없음|“가입하시겠어요?” → **가입 유도**|
|잔액이 부족|“충전하시겠어요?” → **충전 유도**|

`IllegalArgumentException` 하나로는 이 둘을 **구분할 수 없다.** 호출자가 “계좌가 없어서야? 잔액이 부족해서야?“를 알려면 — **실패에 이름이 있어야 한다.**

## 1-2. 이름이 있으면 콕 집어 잡는다

```java
catch (InsufficientBalanceException e) {  // "잔액 부족"만 콕 집어서
    // 충전 유도 로직
}
```

실패에 이름(타입)이 있으면, catch로 **그 실패만 골라서** 잡고 다르게 처리할 수 있다. 이게 커스텀 예외의 값이고, **Spring·실무가 도메인 예외를 만드는 이유**다.

> 핵심: 커스텀 예외 = **의미 있는 이름을 가진 예외**. 호출자가 실패를 구분해 다르게 대응하게 해준다.

# PART 2 — 새 문법 ①: 커스텀 예외 클래스 (상속 + super)

오늘 만든 예외 클래스 (둘 다 똑같은 패턴):

```java
public class AccountNotFoundException extends RuntimeException {
    public AccountNotFoundException(String message) {
        super(message);
    }
}
```

세 줄을 해부한다. **여기 두 개념이 처음 등장한다 — 상속과 super.**

## 2-1. `extends RuntimeException` — 상속(inheritance), 너의 첫 상속

- `extends`는 **“~를 물려받는다”**는 뜻이다. `AccountNotFoundException`은 `RuntimeException`이 가진 모든 기능(메시지 저장, 스택 추적 등)을 **그대로 물려받고**, 거기에 **이름만 내 도메인 것**으로 단다.
- 즉 커스텀 예외는 사실상 **“의미 있는 이름을 새로 붙인 예외 한 종류”**다. 바닥부터 새로 만드는 게 아니라, 이미 있는 예외를 물려받아 이름만 바꾸는 것.
- 이게 객체지향의 **상속**이다. 앞으로 수없이 만난다. (오늘이 첫 상속이다.)

## 2-2. `super(message)` — 부모 생성자 호출

- `super`는 **부모(여기선 RuntimeException)**를 가리킨다. `super(message)`는 **부모의 생성자를 불러서** 내가 받은 `message`를 부모에게 넘긴다.
- 왜 넘기나? **그래야 나중에 `e.getMessage()`로 그 메시지를 꺼낼 수 있다.** 메시지를 저장하는 기능은 부모(RuntimeException)가 갖고 있으니, 부모에게 맡기는 거다.
- 짝 개념: 예전에 Account에서 `this.id = id`로 *내* 필드에 저장했다. 이번엔 `super(message)`로 *부모*에게 넘긴다. 생성자가 값을 받아 처리한다는 점은 같고, 대상이 부모라는 게 다르다.
- **만약 super(message)를 안 쓰면?** 메시지가 부모에게 안 넘어가서, `e.getMessage()`가 `null`을 돌려준다 → 잡아도 “왜 실패했는지” 메시지가 안 나온다.

## 2-3. ★ checked vs unchecked — 왜 RuntimeException인가 (면접 단골)

왜 `extends Exception`이 아니라 `extends RuntimeException`인가? 이게 자바 예외의 가장 중요한 갈림길이다.

|구분  |extends Exception                      |extends RuntimeException|
|----|---------------------------------------|------------------------|
|이름  |**checked** 예외                         |**unchecked** 예외        |
|컴파일러|try/catch나 throws를 **강제** (안 하면 컴파일 거부)|**강제 안 함** (잡고 싶을 때만 잡음)|
|쓰는 곳|복구 가능한 외부 문제(파일 없음 등)                  |도메인/로직 실패               |

- 실무 **도메인 예외는 거의 unchecked**다. 이유: checked로 만들면 모든 호출 지점에서 try/catch를 강제당해 코드가 지저분해지고, 도메인 실패는 보통 **바깥에서 일괄 처리**(글로벌 핸들러)하기 때문.
- → 그래서 `RuntimeException`을 상속했다.
- **면접에서 “checked와 unchecked 차이가 뭐예요?“는 단골 질문이다.** 위 표를 네 말로 설명할 수 있어야 한다.

## 2-4. public class = 파일명 (1일차 규칙)

`public class AccountNotFoundException`이면 파일명은 반드시 `AccountNotFoundException.java`. 그래서 예외 2개는 **각자 자기 파일**이다. (이제 mini-ledger 파일이 5개로 늘었다.)

# PART 3 — 새 문법 ②: try/catch

```java
try {
    bank.transfer("a", "z", 500);          // 위험한 호출 (예외 던질 수 있음)
} catch (AccountNotFoundException e) {       // 그 예외가 터지면 여기로 점프
    System.out.println("잡았다: " + e.getMessage());
}
```

## 3-1. 동작 원리

- `try { }` 안에는 **예외를 던질 수 있는(위험한) 코드**를 넣는다.
- 거기서 예외가 터지면, 프로그램이 **죽는 대신** `catch (예외종류 e) { }` 안으로 **점프**한다.
- catch가 예외를 받아 처리하면, 프로그램은 죽지 않고 계속 진행한다.

## 3-2. `catch (AccountNotFoundException e)`의 `e`

- catch는 “이 종류 예외를 잡아서, **`e`라는 이름으로 받겠다**“는 뜻이다.
- 이 `e`가 있어야 다음 줄 `e.getMessage()`에서 그 예외 객체를 쓸 수 있다.
- **`e`를 빼먹으면**(`catch (AccountNotFoundException)`) → 받을 변수가 없어서 `e.getMessage()`의 `e`가 어디서 왔는지 자바가 모른다 → 컴파일 에러. (오늘 실제로 한 번 빠뜨렸다. 오답노트 O2.)

## 3-3. `e.getMessage()` — PART 2와 연결되는 지점 ★

`e.getMessage()`가 돌려주는 그 메시지가, **PART 2에서 `super(message)`로 부모에게 넘긴 바로 그 메시지**다. 두 개념이 여기서 만난다:

```
new AccountNotFoundException("계좌를 찾을 수 없습니다")
        → super("계좌를 찾을 수 없습니다")   (부모에게 저장)
              → e.getMessage()  →  "계좌를 찾을 수 없습니다"  (꺼냄)
```

이게 super(message)를 쓴 이유의 전부다.

# PART 4 — ★ 절제: 전부 바꾸지 않는다 (오늘의 핵심 판단)

예외를 던지는 곳은 **세 군데**였다. 여기서 중요한 판단: **전부 커스텀 예외로 바꾸면 틀린 배움이다.**

|위치                            |실패        |성격           |처리                                    |
|------------------------------|----------|-------------|--------------------------------------|
|getAccount                    |“계좌 없음”   |**도메인 실패**   |→ `AccountNotFoundException` (바꾼다)    |
|withdraw `balance < amount`   |“잔액 부족”   |**도메인 실패**   |→ `InsufficientBalanceException` (바꾼다)|
|deposit·withdraw `amount <= 0`|“음수 금액 입력”|**그냥 잘못된 인자**|→ `IllegalArgumentException` (안 바꾼다)  |

## 왜 amount <= 0은 안 바꾸나?

- `amount <= 0`은 **도메인 개념(계좌·잔액)의 실패가 아니라, 그냥 잘못된 값이 들어온 것**이다. “음수로 출금”은 비즈니스 실패가 아니라 입력 오류다.
- JDK의 `IllegalArgumentException`이 **이미 이 상황에 정확히 맞는 예외**다. 굳이 새로 만들 이유가 없다.
- **레슨: 커스텀 예외는 남발하는 게 아니다.** JDK 예외가 딱 맞으면 그걸 쓰고, **실패가 네 도메인 개념일 때만**(계좌 없음·잔액 부족) 직접 만든다. 이 절제가 좋은 설계다.

# PART 5 — ★ throw는 깊은 곳, catch는 바깥 (설계 원칙)

오늘 받은 질문: *“try/catch를 왜 Main에 붙여? 다른 파일이 아니라.”* 이게 중요한 원칙이다.

## 5-1. 역할 분리

- **Account / Bank** = 문제가 생기면 **“신고”**하는 쪽 → `throw`. (“나 이거 못 해!”)
- **Main** = 그 기능을 **불러 쓰는** 쪽 = 실패하면 **뭘 할지 결정**하는 쪽 → `try/catch`.

## 5-2. 왜 throw와 catch를 같은 파일에 안 넣나

만약 `withdraw`(Account 안)가 잔액 부족을 **자기가 직접 try/catch로 처리**해버리면:

- withdraw가 “잔액 부족이면 뭘 할지”를 자기 맘대로 정하는 거다.
- 근데 그건 **호출자마다 다르다** — 누구는 “충전 유도”, 누구는 “그냥 취소”, 누구는 “재시도”.
- withdraw는 그걸 **모른다.** 알 수도 없다. **결정은 부르는 쪽(바깥) 몫이다.**

→ 그래서 **던지기는 깊은 곳(Account/Bank)에서, 잡기는 결정할 수 있는 바깥(Main)에서.** 이게 실무 예외 처리의 기본 원칙이다.

(나중에 Spring 가면 이 “바깥”이 **Controller나 글로벌 예외 핸들러**가 된다. 지금 Main이 그 자리의 씨앗이다.)

## 5-3. 비유

Account/Bank는 **“불이야!” 외치는 화재경보기**다. Main은 그 경보를 듣고 **대피할지 소화기 들지 결정하는 사람**이다. 경보기 안에 대피 계획을 넣지 않는다 — 듣는 사람이 상황 보고 정한다.

# PART 6 — ②③ 최종 증명 (돈 증발 스레드 종료)

해피패스(1100/4500) 아래에 붙인 코드:

```java
        // ② 없는 계좌로 이체 → 돈 증발 없는지 증명
        System.out.println("--- 이체 전 a 잔액: " + bank.getAccount("a").getBalance());
        try {
            bank.transfer("a", "z", 500);
        } catch (AccountNotFoundException e) {
            System.out.println("잡았다: " + e.getMessage());
        }
        System.out.println("--- 이체 후 a 잔액: " + bank.getAccount("a").getBalance());

        // ③ 잔액 부족 이체
        try {
            bank.transfer("a", "b", 999999);
        } catch (InsufficientBalanceException e) {
            System.out.println("잡았다: " + e.getMessage());
        }
```

## 6-1. ② 없는 계좌 이체 — 돈 증발 최종 증명 ★

1. 이체 **전** a 잔액 출력 → 1100
1. `transfer("a", "z", 500)` — “z”는 없는 계좌 → getAccount(toId)에서 `AccountNotFoundException` 던짐
1. catch가 잡아서 메시지 출력 → 프로그램 안 죽음
1. 이체 **후** a 잔액 다시 출력 → **여전히 1100** (600 아님!)

**4번이 오늘의 핵심이다.** 없는 계좌로 500원 이체가 실패했는데 **a 돈이 안 줄었다.** 왜? transfer가 `getAccount(from) → getAccount(to) → withdraw → deposit` 순서라서, **to(“z”) 검증에서 먼저 터져 withdraw까지 못 갔기 때문**이다. (6/23에 짠 안전한 순서가 여기서 빛을 발한다.)

→ 이게 **“돈 증발 없음”을 프로그램이 직접 증명**한 거다. 내가 “1100”이라고 적는 게 아니라(6/23 M16의 교훈), **프로그램이 계산해서 1100을 보여준다.** 6/21부터 끌고 온 “돈 증발” 스레드가 여기서 닫혔다.

## 6-2. 실제 출력

```
1100
4500
--- 이체 전 a 잔액: 1100
잡았다: 계좌를 찾을 수 없습니다       ← AccountNotFoundException 잡힘 ✓
--- 이체 후 a 잔액: 1100             ← ★ 600 아니고 1100 = 돈 증발 없음 ✓
잡았다: 잔액이 부족합니다             ← InsufficientBalanceException 잡힘 ✓
```

# PART 7 — 디버깅 케이스 (증상 / 진단 / 근본원인 / 해결 / 교훈)

오늘 코드는 맞았지만 터미널에서 사고가 여러 번 났다. 이게 복기 가치가 크다.

## 케이스 1 — ★ 실제 파일이 안 바뀌어 있었다 (chat-vs-file 괴리)

- **증상:** `grep`을 쳤더니 Bank.java는 여전히 `IllegalArgumentException`, Account.java는 `IllegalStateException`. **분명히 채팅에서 바꿨는데** 옛날 예외 그대로였다.
- **진단:** 채팅에 *보여준* 수정이 **실제 파일에는 저장된 적이 없었다.**
- **근본원인:** **채팅에 코드를 쓰는 것 ≠ 파일에 저장하는 것.** VPS의 실제 파일은 옛 버전 그대로였다.
- **해결:** `sed -i`로 해당 throw 줄만 정확히 교체 (한 줄씩, 나머지 코드 안 건드림).
- **교훈:** **내가 고쳤다고 *생각하는* 것과 파일의 *실제* 상태는 다르다.** 항상 `grep`/`cat`으로 실제 파일을 확인하라. 컴파일러와 grep이 진실이다.

## 케이스 2 — heredoc 붙여넣기 깨짐 (2회)

- **증상:** `cat > Main.java << 'EOF'` 했는데 끝이 `EOF }   }   System.out.println("잡았다...` 처럼 엉켰다. 두 번 연속.
- **진단:** 폰에서 여러 줄 heredoc을 붙이다 입력이 뒤섞여 EOF 종료가 깨졌다.
- **근본원인:** 모바일 키보드의 멀티라인 붙여넣기 깨짐. (6/23 PART 4와 똑같은 함정.)
- **해결:** 블록을 통째로 다시 깔끔히 붙여넣고, 직후 컴파일로 검증.
- **교훈:** heredoc 끝에 EOF가 코드와 엉켜 보이면 깨진 거다. 다시 깔끔히. 붙여넣기 직후엔 항상 검증.

## 케이스 3 — 둥근 따옴표 (채팅 아티팩트)

- **증상:** 채팅에 붙인 코드의 따옴표가 `"계좌를..."`(둥근)으로 보여 컴파일 에러 걱정.
- **진단:** **채팅 입력 경로에서 따옴표가 변환된 것**이고, VPS의 실제 파일은 곧은 `"`였다.
- **근본원인:** 채팅에 붙이는 과정의 렌더링 아티팩트. 실제 파일과 무관.
- **해결:** `javac`가 증명 — 에러 0개로 컴파일됨.
- **교훈:** **채팅에 보이는 걸로 판단하지 마라. 컴파일러가 유일한 진실이다.** 눈으로 따옴표 둥근지 곧은지 다투지 말고 `javac`를 믿어라.

## A-1. 체크박스 토글이 새로고침해도 안 풀리던 버그

- **증상:** 대시보드 마일스톤 체크박스를 (비로그인으로) 눌렀더니, 창을 닫고 새로고침해도 그 상태(진행률 %)가 그대로 남아 있었다.
- **원인:** 토글 상태가 브라우저의 **localStorage**에 저장되고 있었다. localStorage는 같은 브라우저에선 새로고침해도 안 지워진다. (정직하게: 클로드가 처음엔 “새로고침하면 풀린다”고 했는데 **그건 틀린 설명이었다** — localStorage 동작을 잘못 말했다. 네가 정확히 잡았다.)
- **핵심 구분:** 그래도 그건 **네 브라우저에만** 저장된 거다. GitHub의 진짜 데이터(roadmap.json)는 누구도 못 바꾼다. 공개 페이지는 읽기 전용이고, 진짜 진행률은 너만(push) 바꾼다.
- **해결:** 토글을 **세션 한정 미리보기**로 바꿨다. 이제 눌러도 저장 안 되고, 새로고침하면 roadmap.json 기준으로 초기화된다.
- **★ 교훈 (PART 7과 같은 결):** **“그럴 것이다”라고 가정하지 말고 실제 동작을 확인하라.** 이 수정도 Playwright로 “토글 → 새로고침 → 초기화”를 직접 돌려 검증했다. Bank/Account가 실제론 안 바뀌어 있던 PART 7 케이스 1과 **똑같은 교훈** — 가정 말고 검증.

## A-2. cron 로그 폭주

- **증상:** `.cron.log`에 “갱신 완료”가 수천 줄 쌓였다. 1분마다 도는 cron이 매번 빌드를 돌리고 그 로그를 찍고 있었다.
- **원인:** update.py가 idle(새 작업기록 없음)일 때도 build.py를 돌려 로그를 남겼다.
- **해결:** update.py를 고쳐서 **실제 변경(새 작업기록·done·파일 변화)이 있을 때만** 로그를 남기게 했다. idle이면 완전히 침묵.
- **★ 교훈 (6/24 멱등성 재등장):** 자동 반복 작업은 **입력이 없으면 아무 효과도, 흔적도 없어야** 한다. 같은 상태로 1440번 돌아도 결과가 한 번과 같아야 한다. 이게 **멱등성**이고, 어제 git 자동화에서 만난 그 개념이 오늘 cron 로그에서 다시 나왔다. (Layer 1·2의 멱등성 키로 또 이어진다.)

> 두 버그 모두 **“실제 상태를 확인하라”**(A-1)와 **“멱등성”**(A-2)이라는, 오늘 본문에서 이미 나온 원칙의 반복이다. 도구를 고치면서도 같은 교훈을 두 번 더 박은 셈이다.

## 9. 개념 사전

- **커스텀 예외:** 의미 있는 이름을 가진 예외. 호출자가 실패를 구분해 다르게 대응하게 함.
- **상속 (inheritance) / `extends`:** 부모 클래스의 기능을 물려받음. 커스텀 예외 = RuntimeException을 물려받아 이름만 바꾼 것. (오늘 첫 상속)
- **`super(message)`:** 부모 생성자 호출. 메시지를 부모에게 넘겨야 `e.getMessage()`로 꺼낼 수 있음.
- **checked vs unchecked:** checked(extends Exception)=컴파일러가 try/catch 강제. unchecked(extends RuntimeException)=강제 안 함. 도메인 예외는 보통 unchecked. (면접 단골)
- **try/catch:** try에 위험한 코드, 터지면 죽는 대신 catch로 점프.
- **`catch (Type e)`의 e:** 잡은 예외를 받는 변수. 없으면 e.getMessage() 못 씀.
- **`e.getMessage()`:** super(message)로 넘긴 메시지를 꺼냄. (두 개념의 연결점)
- **throw/catch 분리:** 던지기는 깊은 곳(Account/Bank), 잡기는 결정 가능한 바깥(Main). Spring에선 바깥=Controller/글로벌 핸들러.
- **도메인 예외 절제:** JDK 예외가 맞으면(잘못된 인자) 그걸 쓰고, 도메인 개념 실패일 때만 커스텀.
- **heredoc 함정:** 폰 멀티라인 붙여넣기는 깨질 수 있음. 직후 grep/cat 검증.
- **localStorage (부록 A-1):** 브라우저가 데이터를 저장하는 곳. 같은 브라우저에선 새로고침해도 안 지워짐. “공개 페이지”여도 클라이언트(브라우저) 상태는 따로 남을 수 있음 — 가정 말고 실제 동작 확인.
- **멱등성 — 재등장 (부록 A-2):** 같은 작업을 여러 번 해도 결과가 한 번과 같음. 자동 반복(cron)은 입력 없으면 흔적도 없어야 함. 6/24에 이어 두 번째. Layer 1·2 멱등성 키로 이어짐.

## 10. 비유 모음

- **커스텀 예외 = 실패에 이름표:** “뭔가 잘못됨”이라는 익명 신고 대신, “계좌 없음”·“잔액 부족”이라고 이름표를 단 신고. 받는 쪽이 골라서 대응한다.
- **상속 = 기성품에 라벨 갈기:** RuntimeException이라는 기성 예외에 내 도메인 이름표만 새로 붙인 것. 바닥부터 안 만든다.
- **super(message) = 부모에게 짐 맡기기:** 메시지 보관 기능은 부모(RuntimeException)가 가졌으니, 내 메시지를 부모에게 맡긴다. 나중에 찾을 때(getMessage) 부모가 꺼내준다.
- **try/catch = 안전망:** 외줄(위험한 호출)에서 떨어져도(예외) 바닥(프로그램 죽음)에 안 닿게 받아주는 그물.
- **throw vs catch = 화재경보기 vs 대피하는 사람:** 경보기는 “불이야”만 외치고(throw), 대피 방법은 듣는 사람이 정한다(catch). 경보기에 대피계획을 안 넣는다.

## 13. 회고 — 정직하게

오늘 코드는 다 맞았고 푸시도 됐다. **근데 그게 핵심이 아니다.** 오늘의 가장 큰 데이터는 **“내가 직접 한 게 거의 없다”**는 사실이다.

무엇을 스킵했나, 구체적으로:

- 예외 클래스 3줄(상속·super) — **내가 줬다.** 너는 손으로 안 썼다.
- try/catch 틀 — **내가 줬다.** 너는 빈칸 2개만 채웠다.
- Main 전체 — heredoc **복붙**으로 넘겼다.
- 타이핑한 분량이 거의 없다. 본 건 많고, 친 건 적다.

왜 위험한가: **문법은 추론으로 안 나온다. 손으로 쳐야 손에 붙는다.** 눈으로 본 코드는 사흘이면 증발한다. 오늘 “상속”과 “super”를 봤지만, 네 손가락은 그걸 친 적이 없다. 그래서 지금 머리에 안 남은 거다 — 네 느낌이 정확하다.

채찍: **“시간 없었다”가 핑계가 되는 순간, 그게 습관이 된다.** 9개월 뒤 면접에서 “왜 RuntimeException을 상속했어요?“를 물으면, 복붙한 기억으론 못 버틴다. 직접 짠 사람만 답한다. 오늘 너는 답할 재료를 안 만들었다. **그 빚을 위 12번 숙제로 갚아라.** 안 갚으면 오늘은 그냥 날아간 날이다.

다만 — 오늘이 무의미하진 않다. **올바른 설계 판단(절제: amount<=0은 안 바꾼다)을 이해했고, 돈 증발 스레드를 닫는 그림을 봤다.** 그리고 이 회고를 읽고 있다는 것 자체가, 네가 그걸 안다는 증거다. 안다는 건 고칠 수 있다는 거다.

다음엔: 시간이 없으면 **차라리 ②만 하고 직접 쳐라.** 빈칸 채우기로 양을 늘리지 마라. 양보다 “내 손으로”가 백배 낫다. 그리고 증분 4 — enum(TransactionType) + Transaction + 이력(감사 로그의 씨앗)으로 간다.

> **한 줄:** 오늘은 코드가 완성된 날이 아니라, **‘내가 직접 안 했다’는 걸 정직하게 마주한 날**이다. 이 파일을 덮고 빈 화면에서 다시 쳐야 — 그때 비로소 오늘이 끝난다.
