## 5. 코드 — 스캐폴드 해제 (오늘의 본체, 그러나 가장 쉬웠던 부분)

nano로 열어 주석 블록 각 줄의 `//    `를 벗겼다. **[업데이트: 아래는 해제 후 Main.java 전문이다 — 발췌 아님, 커밋 `76c41a0`에 실린 최종 상태.]**

```java
import java.io.IOException;
import java.net.ServerSocket;
import java.net.Socket;

public class Main {
  public static void main(String[] args){
    // You can use print statements as follows for debugging, they'll be visible when running tests.
    System.out.println("Logs from your program will appear here!");

    ServerSocket serverSocket = null;
    Socket clientSocket = null;
    int port = 6379;
    try {
      serverSocket = new ServerSocket(port);
      // Since the tester restarts your program quite often, setting SO_REUSEADDR
      // ensures that we don't run into 'Address already in use' errors
      serverSocket.setReuseAddress(true);
      // Wait for connection from client.
      clientSocket = serverSocket.accept();
    } catch (IOException e) {
      System.out.println("IOException: " + e.getMessage());
    } finally {
      try {
        if (clientSocket != null) {
          clientSocket.close();
        }
      } catch (IOException e) {
        System.out.println("IOException: " + e.getMessage());
      }
    }
  }
}
```

(설명용 `//` 주석 두 줄 — `// Since the tester...`와 `// Wait for connection...` — 은 코드가 아니라 안내라 남겨둬도 무관하다. 벗겨낸 건 `//    ` 접두사가 붙어 **코드를 막고 있던** 줄들이다.)

### 두 줄에 멈춰 새긴 것

**`serverSocket = new ServerSocket(port)` — 포트를 연다.**
7/6에 “마법 목록 ①”으로 적었던 그것 — *Tomcat이 8080을 열고 대기* — 의 아래층이다. mini-ledger·ledger-api 내내 Spring이 대신 열어주던 그 행위를, 오늘 처음 **내 코드 한 줄**이 한다. 6379는 Redis의 표준 포트다.

**`clientSocket = serverSocket.accept()` — 기다린다. 이게 서버의 심장이다.**
7/6 Q1에서 “서버의 존재 이유는 기다림”이라고 **글로** 답했었다. 오늘 그 기다림이 **코드 한 줄**로 처음 보였다. 이 줄에 도달하면 프로그램이 **멈춘다** — 클라이언트가 접속해 올 때까지. 그게 **blocking**이고, 서버가 “떠 있다”는 말의 실제 의미다. CPU를 태우며 도는 게 아니라, 이 줄에서 조용히 멈춰 기다리는 것.

`setReuseAddress(true)`는 테스터가 프로그램을 자주 재시작해도 “Address already in use”가 안 나게 하는 것(주석이 그렇게 설명). 7/7·7/12에 서버 내리고 포트 빠지는 시차를 겪었던 그 문제의, 코드 레벨 예방책인 셈이다.

## 6. 마찰 ④ — ★ 로컬은 통과했는데 원격이 실패했다 (오늘의 진짜 교훈)

해제한 코드가 로컬에서 컴파일됐으니(§4 끝) push했다:

```
root@ledger:~/codecrafters-redis-java# git add . && git commit -m "bind to port 6379 (stage 1)" && git push origin master
[master eb0f749] bind to port 6379 (stage 1)
 2 files changed, 23 insertions(+), 27 deletions(-)
```

CodeCrafters가 원격 빌드를 돌렸고 — **실패했다.** **[업데이트: 원격 빌드 로그 중 실패 원인이 드러나는 핵심 구간 전문.]**

```
remote: [build] > [ERROR] COMPILATION ERROR :
remote: [build] > [ERROR] invalid source release 21 with --enable-preview
remote: [build] >   (preview language features are only supported for release 25)
remote: [build] > [ERROR] Failed to execute goal org.apache.maven.plugins:maven-compiler-plugin:3.13.0:compile (default-compile) on project codecrafters-redis: Compilation failure
remote: [build] > [ERROR] invalid source release 21 with --enable-preview
remote: [build] > [ERROR]   (preview language features are only supported for release 25)
remote: [build] > [ERROR] -> [Help 1]
remote: [build] Error: process "/bin/sh -c .codecrafters/compile.sh" did not complete successfully: exit code: 1.
remote: [build] Build failed. Check the logs above for the reason.
```

**여기가 오늘 가장 값진 지점이다.** 로컬 `mvn -q compile`은 조용히 통과했는데(§3 끝), 원격은 같은 코드를 거부했다. 왜?

**원인:** pom.xml에 내가 못 본 게 하나 더 있었다. grep으로 위치를 찍었다:

```
root@ledger:~/codecrafters-redis-java# grep -n "enable-preview\|compilerArgs\|release" pom.xml
24:                    <compilerArgs>
25:                        <arg>--enable-preview</arg>
26:                    </compilerArgs>
```

`--enable-preview`는 **release 25에서만** 되는 “미리보기 기능 켜기” 플래그다. §3에서 25→21로 낮췄지만 이 플래그는 그대로 남아서, “21인데 preview 켜라”는 모순이 됐다.

**그런데 왜 로컬은 통과했나?** — 이게 핵심 통찰이다. 로컬 JDK 21은 preview 기능을 실제로 안 쓰니 그 플래그를 **무해하게 흘려보냈다.** 반면 CodeCrafters 빌드 환경은 (원래 25용이라) 그 플래그를 **엄격히 검사**해서 모순을 잡아냈다. 즉:

> **로컬 컴파일 성공이 원격 빌드 성공을 보장하지 않는다.** 같은 코드, 같은 pom.xml인데 환경(JDK 버전, 빌드 엄격도)이 다르면 결과가 갈린다.

이건 실무의 핵심 함정 하나다 — “내 컴퓨터에선 되는데요(works on my machine)”. 오늘 그걸 **책이 아니라 실패 로그로** 봤다. ledger-api는 로컬 = 실행 환경이 같아서(둘 다 이 VPS) 이런 어긋남이 안 났는데, CodeCrafters는 **로컬(내 VPS)과 채점(그들 CI)이 분리**돼 있어서 처음 드러난 것.

**해결 — compilerArgs 블록 3줄 통째 제거:**

```
root@ledger:~/codecrafters-redis-java# sed -i '/<compilerArgs>/,/<\/compilerArgs>/d' pom.xml && grep -n "enable-preview\|compilerArgs\|release\|source\|target" pom.xml
12:        <maven.compiler.source>21</maven.compiler.source>
13:        <maven.compiler.target>21</maven.compiler.target>
14:        <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>
```

sed 범위 삭제(`/<compilerArgs>/,/<\/compilerArgs>/d` = 여는 태그부터 닫는 태그까지)로 24~26줄이 사라졌다. grep 결과에 **enable-preview·compilerArgs가 아예 안 나오고** source/target 21만 남았다 = 정확히 그 블록만 제거됨. Stage 1 코드는 preview 기능을 안 쓰니 이 블록 자체가 불필요했다 — 삭제가 정답. 로컬 재확인:

```
root@ledger:~/codecrafters-redis-java# mvn -q compile
root@ledger:~/codecrafters-redis-java#
```

## 11. 회고

코드는 가장 쉬웠고, 그 코드를 돌아가게 만드는 게 오늘의 전부였다. 스캐폴드 주석 풀기는 10분도 안 걸렸다. 그런데 그 앞뒤로 Maven 설치, 자바 버전 불일치, 디렉토리 착각, preview 플래그까지 넷이 줄줄이 걸렸고 — 그걸 하나씩 진단하고 뚫는 게 오늘 시간의 대부분이었다. 이게 “만들어서 안다”의 민낯이다. 튜토리얼은 “주석을 푸세요”까지만 말하고, 그 사이에 낀 네 개의 벽은 안 알려준다. 그 벽을 직접 넘은 게 오늘 남는다.

가장 값진 건 마찰 ④였다. 로컬에서 `mvn compile`이 조용히 통과했을 때 나는 끝난 줄 알았는데, CodeCrafters가 같은 코드를 거부했다. 원인을 파보니 preview 플래그 하나 — 로컬 JDK21은 그걸 안 쓰니 흘려보냈고, 원격은 엄격히 검사해 잡았다. “내 컴퓨터에선 되는데요”라는 그 유명한 함정을, 밈으로가 아니라 실패 로그로 봤다. ledger-api에선 로컬과 실행이 같은 VPS라 이런 어긋남이 원천적으로 없었는데, CodeCrafters는 내 VPS(로컬)와 그들 CI(채점)가 분리돼 있어서 처음 드러났다. 로컬 통과는 필요조건이지 충분조건이 아니다 — 이 한 문장이 오늘의 핵이다.

정직하게 남길 것 하나 더. cd 스코프 실수가 세 번째 나왔다(7/12, 7/14, 오늘). 원인은 조금씩 달라도 증상은 하나 — 내가 지금 어느 폴더에 있는지 확인 안 하고 명령을 친다는 것. 세 번이면 우연이 아니라 절차의 구멍이다. 빌드·실행 명령 앞엔 cd를 붙여서 치는 걸로 못 박는다. 그리고 update.py의 정체도 오늘에야 완전히 잡혔다 — 매분 크론으로 도는 자동봇. 7/19에 나를 어리둥절하게 했던 “커밋 두 개”와 “nothing to commit”이 이 한 줄로 다 설명됐다. 모르고 쓰던 자동화의 뚜껑을 열어본 셈이다.

m106이 닫혔다. 7월 관문은 2주 앞당겨 사실상 완주했고, 남은 건 월말 점검뿐이다. 오늘 처음으로 Spring 없이 포트를 열었다 — 6379에서 `accept()`가 클라이언트를 기다리다 받아냈다. 다음은 Stage 2, RESP 프로토콜의 첫 발(PING에 PONG). 서버가 “듣기”에서 “대답하기”로 넘어가는 자리다. 그 전에 밀린 복기부터 청산하는 게 순서일 수도 있고 — 그건 다음 세션 컨디션 보고 정한다.

> **한 줄:** Spring 없이 6379를 처음 직접 열어 Stage 1 통과(accept()가 테스터를 받아냄) — 코드 10분, 그 코드를 돌아가게 한 환경 마찰 4연쇄가 본체였고, 그중 “로컬 통과 ≠ 원격 통과”(preview 플래그가 CI에서만 걸림)를 실패 로그로 배운 게 오늘의 핵. m106 완결 = 7월 사실상 완주.
