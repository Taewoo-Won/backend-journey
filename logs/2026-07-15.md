## 1. ErrorResponse record — 두 번째 record, 태우가 직접

### 작성한 풀코드 (`ErrorResponse.java` 전체)

```java
package com.ledger.api;

public record ErrorResponse(String code, String message) {}
```

7/7의 TransferRequest 이후 **두 번째 record**다. 그때는 템플릿을 받았지만 오늘은 자리(빈칸)만 짚어주고 태우가 채웠다. 1일차 규칙(public record 이름 = 파일명) 그대로.

### 교정 2곳 (정직 기록)

|#|첫 시도                                           |문제                                                         |교정              |
|-|-----------------------------------------------|-----------------------------------------------------------|----------------|
|①|`public record ErrorResponse(, String Message)`|**첫 필드 누락** — 콤마 앞이 빈칸(code가 빠짐)                           |`String code` 채움|
|②|`String Message`                               |**대문자로 시작** — 자바 관례는 필드명 소문자(대문자는 클래스명과 혼동, 컴파일은 되나 스타일 오류)|`String message`|

②는 TransferRequest의 `fromId`, `toId`가 전부 소문자로 시작했던 것과 같은 규칙 — 7/7에 이미 본 패턴이다. `cat` + `compileJava`로 `public record ErrorResponse(String code, String message) {}` + BUILD SUCCESSFUL 확인.

## 2. ★ code가 왜 필요한가 — 오늘의 진짜 본체

태우의 반응이 정확했다: **“String Message는 유추 가능했는데 String Code는 뜬금없다.”** 이 반응 자체가 중요한 신호다 — message는 “에러 메시지”라는 뜻에서 자연히 나오는데, code는 뜻이 아니라 **설계 결정**에서 나오기 때문이다.

### message 하나로는 왜 안 되나

어제(7/14 §6) 관찰한 상태: 세 실패(없는 계좌·잘못된 금액·잔액부족)가 **전부 400**이고, 본문 메시지도 한글 문장이다. 이 API를 쓰는 앱이 “잔액 부족이면 충전 화면 띄우기”처럼 **경우별로 다르게 반응**하려면, 지금 유일한 방법은:

```java
if (message.contains("잔액이 충분하지")) { 충전화면(); }
```

**이게 왜 끔찍한가 (어제 Q2에서 이미 결론):**

- 오타 수정 한 번 → 앱이 조용히 깨진다 (에러도 안 남)
- 번역 추가(영어 서비스) → 문장이 바뀌니 전부 깨진다
- 말투 순화(“충분하지 않습니다” → “부족합니다”) → 깨진다

문장은 **사람에게 보여주려고** 바뀌는 게 정상인데, 그 문장에 **기계 로직이 매달려** 있으면 사람 편의와 기계 안정성이 충돌한다.

### code가 그 충돌을 없앤다

|   |`code`                |`message`       |
|---|----------------------|----------------|
|대상 |**기계**(앱 코드)          |**사람**(화면)      |
|예시 |`INSUFFICIENT_BALANCE`|`잔액이 충분하지 않습니다` |
|변경 |**절대 안 바뀜**(계약)       |바뀌어도 됨(번역·순화 자유)|
|스타일|대문자 상수 (6/27 enum과 동일)|자연어 문장          |

앱은 `if (code.equals("INSUFFICIENT_BALANCE"))`로 분기하고, `message`는 화면에 그냥 띄운다. 이제 문장을 아무리 바꿔도 앱은 안 깨진다 — **분기의 기준(code)과 표시물(message)이 분리**됐기 때문이다.

→ **즉 이건 새 자바 문법이 아니라 “왜 필드가 2개여야 하는가”의 답이었다.** code의 정체는 6/27에 enum 만들 때 이미 써본 “대문자, 불변, 기계용 이름” 스타일 그대로다. 낯선 건 문법이 아니라 그게 **설계에서 나온다는 것**이었다.

## 3. 핸들러 3개 수정 — `<String>` → `<ErrorResponse>`

3핸들러 동시 수정이라 EOF heredoc으로 한 번에 덮어썼다(7/7에 배운 이유: nano 저장 실수 위험 제거). 각 핸들러에서 두 곳씩, 총 6곳:

1. 반환 타입: `ResponseEntity<String>` → `ResponseEntity<ErrorResponse>`
1. body: `.body("이체 실패: " + e.getMessage())` → `.body(new ErrorResponse("코드값", e.getMessage()))`

### 완성된 TransferController 전문 (커밋 966c93f)

```java
package com.ledger.api;

import org.springframework.web.bind.annotation.PostMapping;
import org.springframework.web.bind.annotation.RequestBody;
import org.springframework.web.bind.annotation.RestController;
import org.springframework.http.HttpStatus;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.ExceptionHandler;

@RestController
public class TransferController {

    private final Bank bank = new Bank();

    public TransferController() {
        bank.createAccount("a", "이름", 2100);
        bank.createAccount("b", "이름", 3500);
        bank.createAccount("platform", "플랫폼", 0);
    }

    @PostMapping("/transfer")
    public String transfer(@RequestBody TransferRequest request) {
        bank.transfer(request.fromId(), request.toId(), request.amount(), request.idempotencyKey());
        return "a: " + bank.getAccount("a").getBalance()
             + ", b: " + bank.getAccount("b").getBalance()
             + ", platform: " + bank.getAccount("platform").getBalance();
    }

    @ExceptionHandler(InsufficientBalanceException.class)
    public ResponseEntity<ErrorResponse> handleInsufficientBalance(InsufficientBalanceException e) {
        return ResponseEntity.status(HttpStatus.BAD_REQUEST).body(new ErrorResponse("INSUFFICIENT_BALANCE", e.getMessage()));
    }

    @ExceptionHandler(AccountNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleAccountNotFound(AccountNotFoundException e) {
        return ResponseEntity.status(HttpStatus.BAD_REQUEST).body(new ErrorResponse("ACCOUNT_NOT_FOUND", e.getMessage()));
    }

    @ExceptionHandler(IllegalArgumentException.class)
    public ResponseEntity<ErrorResponse> handleIllegalArgument(IllegalArgumentException e) {
        return ResponseEntity.status(HttpStatus.BAD_REQUEST).body(new ErrorResponse("INVALID_AMOUNT", e.getMessage()));
    }
}
```

### code 값 3개 — 왜 이렇게 지었나

|예외                          |code 값                |근거                                                                                       |
|----------------------------|----------------------|-----------------------------------------------------------------------------------------|
|InsufficientBalanceException|`INSUFFICIENT_BALANCE`|클래스명 그대로 (도메인 예외라 이름이 이미 정확)                                                             |
|AccountNotFoundException    |`ACCOUNT_NOT_FOUND`   |클래스명 그대로                                                                                 |
|IllegalArgumentException    |`INVALID_AMOUNT`      |★ **클래스명 안 씀** — JDK 공용이라 “IllegalArgument”는 너무 포괄적. “이 컨트롤러에서 실제로 뭐가 잘못됐나”(금액)를 code에 반영|

세 번째가 7/14 §1의 트레이드오프와 연결된다 — IllegalArgumentException은 JDK 공용이라 이 컨트롤러에서 amount 검증 말고 다른 데서도 터질 수 있다. 그래서 code를 클래스명(`ILLEGAL_ARGUMENT`)이 아니라 **현재 이 자리의 실제 의미**(`INVALID_AMOUNT`)로 좁힌 것. 판정: `@ExceptionHandler`=3, BUILD SUCCESSFUL in 7s. cat으로 전문 재확인(폰 붙여넣기가 화면상 깨져 보였지만 실제 파일은 온전 — 눈으로 전체 확인 후 진행, “가정 말고 확인” 그대로).

## 4. ★ 검증 4연타 — Content-Type이 바뀌는 순간

부팅 고정 절차(grep Started 판정 → 그다음 curl) 지킴. `Started in 4.767s` 확인 후:

|#|요청          |상태 |본문                                                     |Content-Type        |
|-|------------|---|-------------------------------------------------------|--------------------|
|①|없는 계좌(zzz)  |400|`{"code":"ACCOUNT_NOT_FOUND","message":"계좌를 찾을 수..."}` |**application/json**|
|②|amount -100 |400|`{"code":"INVALID_AMOUNT","message":"출금액은 0보다 커..."}`  |**application/json**|
|③|amount 99999|400|`{"code":"INSUFFICIENT_BALANCE","message":"잔액이 충분..."}`|**application/json**|
|④|정상 이체(회귀)   |200|`a: 1080, b: 4500, platform: 20`                       |text/plain          |

**★ 오늘의 관전 포인트가 정확히 찍혔다:** 어제까지 `text/plain`이던 Content-Type이 세 에러 전부 **`application/json`**으로 바뀌었다.

### 이게 왜 중요한가 — 마법 ④의 역방향

7/7에 배운 마법 ④(@RequestBody)는 **JSON → 객체**(역직렬화)였다:

```
curl이 보낸 JSON 텍스트  →  잭슨  →  TransferRequest record
```

오늘은 **정확히 반대 방향**이 처음 가동됐다:

```
new ErrorResponse(...) 객체  →  잭슨  →  JSON 텍스트 + Content-Type: application/json
```

`body()`에 String을 넣으면 잭슨이 할 일이 없어 그대로 text/plain으로 나갔는데(어제까지), **record(객체)를 넣으니 잭슨이 “이건 JSON으로 바꿔야겠다”** 판단해서 직렬화하고 헤더까지 application/json으로 바꾼 것. **같은 마법(잭슨)의 왕복 두 방향을 이제 다 봤다** — 들어올 때(@RequestBody)와 나갈 때(return 객체).

(터미널에 본문이 `계좌를 찾을 수`처럼 끊겨 보인 건 다음 명령이 같은 줄에 이어져 화면상 잘린 것 — 실제 응답이 잘린 게 아니다. code 매칭 정확, message도 도메인 문장 시작 일치.)

### 부수 관찰 — 정상 이체는 아직 text/plain

④(정상, 200)는 여전히 `text/plain`이다. 오늘은 **에러 경로만** 손댔으니 당연한 결과지만, 성공 응답도 구조화하면 API 전체가 일관된 JSON이 된다. **다음 조각 후보**로만 기록(지금 할 필요 없음).

## 5. 별건 — code-server 편집환경 시험 (성공)

### 왜 이걸 했나

폰으로만 타이핑하니 VPS 화면(nano)이 좁아 불편하다는 문제. 태우가 “PC에서 타이핑 → 네이버 내게쓰기 → 폰에 복붙” 안을 냈는데, 여기엔 제약이 걸린다 — **사지방(부대 공용 PC)은 GitHub 로그인이 안 된다**(github.dev·웹에디터 안 죽음). 그래서 대안: **VPS 자체에 브라우저용 VS Code(code-server)를 띄우면** 사지방 브라우저로 내 서버에 접속해 편집 — GitHub 로그인도, 메일 다리도 필요 없다.

### 시험 절차와 판정

```bash
curl -fsSL https://code-server.dev/install.sh | sh   # 4.128.0 설치
code-server --version && ufw status                  # → 4.128.0 / Status: inactive (방화벽 안 막음)
```

설정 수정(`~/.config/code-server/config.yaml`): `bind-addr`을 기본 `127.0.0.1:8080`(로컬 전용) → **`0.0.0.0:8443`**(외부 접속 + 8080은 Tomcat과 충돌 회피). 비번은 자동생성값 유지(grep -c로 줄 존재만 확인, 값은 화면에 안 띄움).

```bash
code-server > code-server.log 2>&1 &
grep "HTTP server listening" code-server.log   # → HTTP server listening on http://0.0.0.0:8443/
```

### 보안 규율

code-server는 **공개 IP에 떠 있는 에디터**라 ledger-api보다 오히려 더 민감하다(누구나 접속 시도 가능). 그래서 강한 비번(자동생성) + **“쓸 때만 켜고 끝나면 내린다”** 규율을 그대로 적용 — 오늘 시험 끝나고 즉시 내렸다.

## 9. 회고

코드는 적었지만 이해가 컸던 날이다. record 한 줄, 타입 교체 세 곳 — 손이 움직인 양은 30분도 안 걸릴 분량이었다. 그런데 시간을 쓴 곳은 “code가 대체 왜 필요하냐”였고, 그 질문이 나온 것 자체가 오늘의 핵심이었다. message는 “에러 메시지”라는 뜻에서 자연히 유추됐는데 code는 안 됐다 — “뜬금없다”는 반응이 정확했던 건, 이게 문법이 아니라 설계라서다. 어제 Q2에서 “문자열 분기는 끔찍하다”를 논리로 알았고, 오늘 그 해법(code)을 직접 만들면서 왜 두 필드여야 하는지를 몸으로 이었다. code=기계용 불변, message=사람용 가변. 이 분리가 오늘 남는 한 줄이다.

눈으로 본 것도 값졌다. Content-Type이 text/plain에서 application/json으로 바뀌는 순간 — body에 String을 넣던 어제와 record를 넣는 오늘의 유일한 차이가, 잭슨이 직렬화를 하느냐 마느냐를 갈랐다. 7/7에 JSON이 record가 되는 걸(마법 ④) 봤는데, 오늘 record가 JSON이 되는 역방향을 봤다. 같은 마법의 왕복을 다 본 셈이다. 이건 설명으로 들은 게 아니라 헤더 한 줄이 바뀐 걸로 확인했다 — 가정 말고 확인, 오늘도 유지됐다.

정직하게 남길 것 둘. 하나 — 부팅 절차를 지켰다. 7/14에 “성급한 curl 재발”을 고정 절차로 못박았는데, 오늘은 grep Started 판정을 먼저 하고 curl로 갔다. 실수를 규칙으로 바꾼 게 하루 만에 지켜졌다. 둘 — record 교정 2곳(첫 필드 누락, 대문자 시작)은 사소하지만, cat으로 전체를 다시 확인하고 넘어간 습관이 폰 붙여넣기 깨짐(화면상)에도 안 흔들리게 해줬다. 숫자(grep=3)만 믿지 않고 눈으로 전문을 본 것.

그리고 오늘은 코드 밖에서도 하나 풀었다. 폰 편집의 불편을 code-server로 우회하는 다리를 놨다 — 사지방 GitHub 로그인 불가라는 제약을 정면으로 받고, VPS에 내 에디터를 띄우는 방식으로 돌아갔다. 방화벽 하나만 남았으니 다음 사지방 때 확인하면 된다. 편집은 브라우저로, 실행·검증은 여전히 폰 Termius로 — 역할이 갈리지만 그게 지금 환경의 최선이고, 근본 해결(블루투스 키보드)은 건의서에 있다.

예외 번역 장이 이걸로 닫혔다. 다음은 완전히 다른 결 — m106, 내 손으로 6379 포트를 여는 것. 7/6에 “마법 목록 ①”로 올렸던 그것(Tomcat이 8080을 연다)의 아래층을, 이번엔 직접. 8월 저수준의 워밍업이고, 이 포폴의 정체성 그 자체다.
